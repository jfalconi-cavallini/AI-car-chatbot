GET /
  Handler:     Reads index.html from disk and returns it as a FileResponse
               with content-type text/html.
  Auth:        None
  Purpose:     Serves the entire frontend application.

POST /chat
  Handler:     ask_gpt()
  Auth:        None
  Content-Type: application/json
  Request Body (Pydantic model UserQuery):
    question:   string  (required) — the user's message
    session_id: string  (optional) — UUID for conversation continuity
  Response:
    {
      "response":   string,   — HTML or plain text response from the AI
      "session_id": string    — UUID to pass back on next request
    }
  Logic:
    1. If session_id not provided, generate new UUID
    2. Initialize or retrieve session history from sessions dict
    3. Append user message to history
    4. Call OpenAI chat completions with history + tools definition
    5. If GPT requests tool call (get_cars):
         - Parse JSON arguments from tool_call
         - Call get_cars() with those arguments
         - Return HTML-formatted car listings
    6. If no tool call:
         - Return GPT's text response
    7. Append assistant response to session history
    8. Return response + session_id

External API: GET CAR_API_URL
  Called by:   get_cars() function inside main.py
  Protocol:    HTTP GET via requests.get()
  Response:    JSON array of car objects
  Car object fields:
    year, make, model, price, mileage,
    exterior_color, interior_color, image_url, link

---- System Prompt ----

  "You are a fast, friendly, professional car dealership assistant.
  Always greet new users with a welcome message.
  Engage briefly, then show cars if the user asks.
  Present cars with year, make, model, price, mileage,
  exterior/interior colors, image, link, and highlight extremes.
  If the user asks for 'any car', show random cars first, then ask
  if they want to refine by make, model, year, price, or mileage.
  Remember previous user preferences during the session.
  Handle vague queries politely and support 'more' to show
  additional cars."

---- Tool / Function Calling ----

  OpenAI's tool calling feature allows the model to decide when to
  invoke a predefined function. The model returns a structured
  tool_call object instead of (or before) generating a text response.

  Tool definition passed to the API:
  {
    "type": "function",
    "function": {
      "name": "get_cars",
      "description": "Retrieve car inventory based on filters",
      "parameters": {
        "type": "object",
        "properties": {
          "make":           { "type": "string" },
          "model":          { "type": "string" },
          "year":           { "type": "integer" },
          "max_price":      { "type": "number" },
          "max_mileage":    { "type": "integer" },
          "exterior_color": { "type": "string" },
          "interior_color": { "type": "string" },
          "limit":          { "type": "integer" }
        },
        "required": []   — All parameters are optional
      }
    }
  }

  tool_choice: "auto"
  — The model autonomously decides whether to call the tool based on
    the user's query. A question like "show me cheap SUVs" triggers
    the tool; a question like "what should I know about buying a car?"
    does not.

---- Conversation History ----

  Full history is included in every API call:
    messages = [system_prompt] + session["history"]
  This gives the model full context of the conversation, enabling it
  to remember user preferences (e.g., "I like blue cars under $20k")
  without re-stating them on every message.

---- Response Types ----

  Text response:   GPT returns a plain text message (greeting, clarification,
                   general automotive advice, etc.)
  Tool response:   GPT requests get_cars() → backend executes it → formats
                   result as HTML → returns HTML string to the frontend

  SECTION 8: CORE ALGORITHMS & BUSINESS LOGIC
--------------------------------------------------------------------------------

---- normalize_color(fancy_name) ----

  Input:   A color string from the car inventory API (e.g., "Midnight Black",
           "Crimson Red", "Pearl White")
  Process:
    1. Convert input to lowercase
    2. Iterate through COLOR_MAP entries
       COLOR_MAP maps variants to standard names:
         "red":    ["red", "crimson", "ruby", "maroon", "wine", "cherry", ...]
         "blue":   ["blue", "navy", "cobalt", "sapphire", "azure", ...]
         "black":  ["black", "midnight", "onyx", "obsidian", "shadow", ...]
         "white":  ["white", "pearl", "ivory", "cream", "snow", ...]
         "silver": ["silver", "gray", "grey", "platinum", "titanium", ...]
         "green":  ["green", "forest", "olive", "emerald", "sage", ...]
         "brown":  ["brown", "tan", "beige", "bronze", "copper", ...]
         "gold":   ["gold", "champagne", "sandy", "yellow", ...]
         "orange": ["orange", "amber", ...]
         "purple": ["purple", "violet", "plum", "lavender", ...]
    3. For each standard color, check if ANY of its variant strings are
       a substring of the input (case-insensitive)
    4. Return the matched standard color name, or "other" if no match
  Output:  Normalized color string (one of the COLOR_MAP keys, or "other")

---- get_cars(make, model, year, max_price, max_mileage, exterior_color,
              interior_color, relax_filters=True, limit=5) ----

  Step 1 — Fetch inventory:
    requests.get(CAR_API_URL) → JSON list of all car dicts

  Step 2 — Normalize colors on all cars:
    For each car: normalize exterior_color, interior_color

  Step 3 — Apply filters (nested apply_filters function):

    Strict mode (strict=True):
      make:           car["make"].lower() == filter_make.lower()
      model:          car["model"].lower() == filter_model.lower()
      year:           car["year"] == filter_year (int comparison)
      max_price:      car["price"] <= max_price
      max_mileage:    car["mileage"] <= max_mileage
      exterior_color: normalized color == filter_color
      interior_color: normalized color == filter_color

    Relaxed mode (strict=False, applied if strict returns 0 results
    AND relax_filters=True):
      make:  filter_make.lower() in car["make"].lower()  (substring)
      model: filter_model.lower() in car["model"].lower() (substring)
      price/mileage/color: same as strict

  Step 4 — Score each car (nested score function):
    For each applied filter, if the car matches: score += 1
    Sort by score descending (most matches first)

  Step 5 — Identify extreme cars and add emoji highlights:
    💰 Cheapest car (lowest price)
    💎 Most expensive car (highest price)
    🛣️  Highest mileage car
    🚗 Lowest mileage car
    🆕 Newest year
    📜 Oldest year

  Step 6 — Format as HTML string:
    For each car in top `limit` results:
      - Title line: "Year Make Model | image_url" (pipe-separated for frontend
        to detect and render as a card)
      - Price, mileage, exterior/interior colors
      - Image <img> tag
      - Hyperlink to listing

  Output: HTML string with car cards OR error message string

---- ask_gpt(user_question, session_id=None) ----

  1. Generate session_id UUID if not provided
  2. Initialize session dict if new: sessions[id] = {"history": []}
  3. Append { "role": "user", "content": user_question } to history
  4. Build messages list: [system_prompt_dict] + history
  5. Call openai.chat.completions.create(
       model="gpt-4o-mini",
       messages=messages,
       tools=[get_cars_tool_definition],
       tool_choice="auto"
     )
  6. Extract response = completion.choices[0].message
  7. If response.tool_calls is not None:
       - Get tool_call = response.tool_calls[0]
       - args = json.loads(tool_call.function.arguments)
       - result = get_cars(**args)  — execute the inventory lookup
       - assistant_reply = result   — HTML-formatted car listings
  8. Else:
       - assistant_reply = response.content  — GPT's text response
  9. Append { "role": "assistant", "content": assistant_reply } to history
  10. Return assistant_reply, session_id

--------------------------------------------------------------------------------
  SECTION 9: SESSION MANAGEMENT
--------------------------------------------------------------------------------

Storage:      In-memory Python dictionary (module-level variable)
Structure:    sessions: dict[str, dict]
Key:          session_id — UUID4 string (e.g., "f47ac10b-58cc-4372-a567-...")
Value:        { "history": [ {role, content}, ... ] }

Lifecycle:
  - Created: First POST /chat request with no session_id (or new UUID)
  - Updated: Every subsequent message in the conversation
  - Destroyed: Server restart / Railway redeploy (no persistence)

Client Responsibility:
  - Client stores the session_id returned in the first response
  - Must include session_id in all subsequent POST /chat requests
  - If omitted, a new session is created (conversation context is lost)

Current Implementation in index.html:
  - The frontend does NOT currently persist session_id between messages
  - Each request is sent without a session_id → new session created each time
  - This means multi-turn context does NOT work in the current frontend
  - Backend supports it fully; frontend implementation is incomplete

Scalability Limitation:
  - In-memory storage doesn't survive server restarts
  - Not cluster-safe (sessions lost if Railway scales to multiple instances)
  - For production: replace with Redis or a database-backed session store

---- JavaScript (inline <script>) ----

  Constants:
    API_URL: Detects environment:
      - localhost → http://localhost:8000/chat
      - Production → /chat (relative, same origin)

  sendMessage():
    1. Read and trim value from #user-input
    2. Return early if empty
    3. Append user message to #messages via appendMessage("user", text)
    4. Clear #user-input
    5. Append loading spinner div to #messages
    6. Scroll to bottom of #messages
    7. fetch(API_URL, { method: "POST", body: JSON.stringify({question: text}) })
    8. Parse response JSON
    9. Remove loading spinner
    10. Call renderBotMessage(data.response)
    11. Scroll to bottom

  appendMessage(role, text):
    - Creates a <div> with class "user-message" or "bot-message"
    - Sets innerText (safe, no HTML injection)
    - Appends to #messages

  renderBotMessage(text):
    - Splits response by newline
    - For each line:
        If line contains "|": split on "|"
          → parts[0] = car title (text before pipe)
          → parts[1] = image URL (text after pipe)
          → Creates a .car-card div with title + <img> tag
        Else:
          → Creates a <div> with innerHTML = line
          (This path allows raw HTML rendering from backend responses)
    - Appends all rendered elements to #messages

  Event Listeners:
    #send-btn "click"  → sendMessage()
    #user-input "keydown" → if Enter key pressed → sendMessage()

  Note: renderBotMessage uses innerHTML for non-pipe lines, which means
  HTML returned from the backend is rendered directly. The car listing
  lines also construct HTML. This is functional but has XSS implications
  if the car inventory API or OpenAI were ever to return malicious content.

--------------------------------------------------------------------------------
  SECTION 14: SECURITY CONSIDERATIONS
--------------------------------------------------------------------------------

Implemented:
  - .env excluded from git via .gitignore
  - API keys not hardcoded in source files
  - Pydantic request body validation (type checking on /chat inputs)
  - Session IDs are UUIDs (not sequential, not guessable)

Vulnerabilities / Gaps:

  XSS (Cross-Site Scripting):
    - renderBotMessage() in index.html sets innerHTML on non-pipe lines
    - Backend returns raw HTML strings from get_cars() formatting
    - If CAR_API_URL or OpenAI response ever contained <script> tags,
      they would execute in the browser
    - Mitigation: Sanitize HTML on frontend with DOMPurify, or use
      textContent / structured JSON responses

  No Authentication:
    - /chat endpoint is public — any user can make unlimited requests
    - No API key or user identity required
    - No CORS restriction — any origin can POST to /chat

  No Rate Limiting:
    - Each POST /chat can trigger OpenAI API calls (costs real money)
    - Malicious user could spam requests and exhaust API quota/budget
    - Mitigation: Add rate limiting middleware (e.g., slowapi for FastAPI)

  In-Memory Sessions:
    - Sessions lost on server restart
    - No session expiry (sessions grow unbounded in long-running processes)
    - No invalidation mechanism

  OpenAI API Key Exposure Risk:
    - If .env is accidentally committed or Railway env vars are leaked,
      the key is exposed
    - Mitigation: Rotate key immediately if exposed; use OpenAI usage limits

  No HTTPS Enforcement:
    - Railway provides HTTPS by default, but no redirect from HTTP is coded

--------------------------------------------------------------------------------
  SECTION 15: KNOWN LIMITATIONS & GAPS

Session Continuity Not Working in Frontend:
  - Backend fully supports session_id for multi-turn context
  - Frontend never stores or re-sends the session_id
  - Every message starts a new session → GPT loses memory between messages
  - Fix: Store data.session_id in a JS variable and include it in each request

No Pagination Implementation:
  - System prompt mentions "more" command for additional results
  - Backend has limit parameter in get_cars()
  - No offset/page tracking is implemented — "more" would just re-fetch
    the same top N results

Hardcoded Limits:
  - Default limit=5 cars returned per query
  - No configurable page size

No Logging:
  - Uses print() statements (not Python logging module)
  - No structured logs, no log levels, no timestamps

No Unit Tests:
  - No test files present
  - No test runner configured

README Discrepancy:
  - README describes a React + Node.js/Express architecture
  - Actual implementation is Python FastAPI + vanilla JS
  - README reflects a planned but unbuilt design

Car API Dependency:
  - Application entirely dependent on external CAR_API_URL being online
  - No caching of inventory (fetched fresh on every request)
  - No fallback if the car API is down

Color Matching:
  - Substring matching can produce false positives
    (e.g., "silver" matching "silverstone gray" or unrelated words)
  - Regex-based matching would be more precise
